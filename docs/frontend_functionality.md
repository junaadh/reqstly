# Frontend Functionality (Phase 3 Prep)

This maps the Stitch screens to frontend behavior and required API contracts.

## 1) Reqstly Login Screen (`/login`)

Features:
- Email/password sign-in form.
- Show/hide password toggle.
- Passkey sign-in option.
- Microsoft Azure OAuth sign-in option.
- "Forgot password" link placeholder.
- Redirect to dashboard on success.

API contracts:
- route: `SUPABASE /auth/v1/token?grant_type=password`
- is protected: `no`
- purpose: Authenticate user with email/password.
- response: Supabase session payload with `access_token` on success.
- https method: `POST`
- other: Called via `supabase-js signInWithPassword`.

- route: `SUPABASE /auth/v1/authorize?provider=azure` (provider OAuth flow)
- is protected: `no`
- purpose: Start Microsoft Azure OAuth login flow.
- response: Redirect URL / OAuth session continuation.
- https method: `GET` (redirect flow via SDK)
- other: Called via `supabase-js signInWithOAuth` with provider `azure`.

- route: `SUPABASE /auth/v1/webauthn/passkey` (unofficial GoTrue route)
- is protected: `no`
- purpose: Start passkey authentication challenge for first-factor passkey login.
- response: `PublicKeyCredentialRequestOptions` payload.
- https method: `POST`
- other: Implemented in frontend; on deployments without this route GoTrue returns `404`.

- route: `SUPABASE /auth/v1/webauthn/passkey/verify` (unofficial GoTrue route)
- is protected: `no`
- purpose: Verify WebAuthn assertion generated by browser ceremony.
- response: Verification payload and/or session material.
- https method: `POST`
- other: Implemented in frontend passkey helper.

- route: `SUPABASE /auth/v1/webauthn/session` (unofficial GoTrue route)
- is protected: `no`
- purpose: Exchange passkey verification payload for session tokens when verify response does not include tokens directly.
- response: Access/refresh token payload.
- https method: `POST`
- other: Implemented as fallback in frontend passkey helper.

## 5) Reqstly Requests List View (`/requests`)

Features:
- Requests table.
- Assignee column (name/email or unassigned).
- Filter controls (status/category/priority).
- Sort control.
- Pagination controls.
- Server-side `q` search (tokenized) across title, description, category, status, priority, and assignee text.
- Accessible action menu semantics on desktop rows (`menu` / `menuitem`).
- Live websocket patch updates (no invalidate/refetch).

API contracts:
- route: `/api/v1/requests?page=<n>&limit=<n>&sort=<sort>&status=<status>&category=<category>&priority=<priority>&q=<term>`
- is protected: `yes`
- purpose: Fetch request list with server-side filtering/sort/pagination/search.
- response: `ApiListEnvelope<SupportRequest>`
- https method: `GET`
- other: `q` is split on whitespace; all terms must match at least one searchable field.

- route: `/api/v1/meta/enums`
- is protected: `yes`
- purpose: Populate status/category/priority filter options.
- response: `ApiEnvelope<RequestEnums>`
- https method: `GET`
- other: Fallback values are present in UI if this fails.

- route: `/api/v1/assignees/suggestions?limit=100&q=<term>`
- is protected: `yes`
- purpose: Suggest assignable users scoped to the signed-in user email domain.
- response: `ApiEnvelope<AssigneeSuggestion[]>`
- https method: `GET`
- other: Ordered by `assignment_count` desc (popular first), then email; supports tokenized narrowing via `q`.

## 16) Realtime Transport (`/ws`)

Features:
- Per-user authenticated websocket stream from Rust backend.
- No full-page invalidate/refetch for request mutations.
- Event-driven UI patching for dashboard stats/recent list, requests list, and request detail/audit timeline.
- Deterministic reconnect resync: active page invalidates only its own dependency key (`dashboard`, `requests:list`, `requests:detail:{id}`).
- Connection status indicator in app shell (`Live` / `Syncing`).

API contracts:
- route: `/ws?token=<supabase_access_token>` (or `Authorization: Bearer <token>`)
- is protected: `yes`
- purpose: Open authenticated realtime stream for request domain events.
- response: WebSocket upgrade + JSON event envelopes.
- https method: `GET` (WebSocket upgrade)
- other: Origin is validated against backend allowlist; messages are versioned (`v: 1`).

- route: `WS event type: request.created`
- is protected: `yes` (server-side recipient filtering by request participants)
- purpose: Notify clients about newly visible request.
- response: `{ request: SupportRequest }`
- https method: `push event`
- other: Used on initial create and on assignment-driven visibility changes so newly added participants get a deterministic insert/count update.

- route: `WS event type: request.patch`
- is protected: `yes`
- purpose: Notify clients about assignment/status/field changes.
- response: `{ request: SupportRequest, changed_fields: string[], previous_status: string }`
- https method: `push event`
- other: Emitted to already-visible participants for deterministic patching and status counter deltas.

- route: `WS event type: request.deleted`
- is protected: `yes`
- purpose: Notify clients to remove deleted request from local UI state.
- response: `{ id: string, status: string }`
- https method: `push event`
- other: Detail page navigates away when active request is deleted.

- route: `WS event type: audit.append`
- is protected: `yes`
- purpose: Append new audit timeline item without refetch.
- response: `{ audit: AuditLog }`
- https method: `push event`
- other: Detail timeline prepends newest event and caps retained history on client.

- route: `WS event type: sync.required`
- is protected: `yes`
- purpose: Signal client to run targeted dependency invalidation for the active view when server requests resync.
- response: `{}`
- https method: `push event`
- other: Client also triggers the same targeted resync after websocket reconnect.

## 11) Reqstly Profile Overview Screen (`/profile`)

Features:
- Display name field (editable in UI).
- Read-only email and user ID.
- Add passkey action (WebAuthn MFA enrollment).
- Save changes persists profile display name through backend API.

API contracts:
- route: `/api/v1/me`
- is protected: `yes`
- purpose: Load profile identity data.
- response: `ApiEnvelope<MeProfile>`
- https method: `GET`
- other: Implemented now.

- route: `/api/v1/me`
- is protected: `yes`
- purpose: Update display name/profile metadata.
- response: `ApiEnvelope<MeProfile>`
- https method: `PATCH`
- other: Validation requires non-empty `display_name` and max length 120.

- route: `SUPABASE /auth/v1/factors`
- is protected: `yes`
- purpose: Enroll WebAuthn MFA factor for current authenticated user.
- response: Factor envelope including `id` for follow-up challenge.
- https method: `POST`
- other: Requires `Authorization: Bearer <access_token>` and `apikey` header. If disabled in GoTrue config, returns `mfa_webauthn_enroll_not_enabled`.

- route: `SUPABASE /auth/v1/factors/{factorId}/challenge`
- is protected: `yes`
- purpose: Request WebAuthn registration challenge (`type=create`).
- response: Challenge envelope with credential options.
- https method: `POST`
- other: Frontend sends `webauthn.rpId` and `webauthn.rpOrigins`.

- route: `SUPABASE /auth/v1/factors/{factorId}/verify`
- is protected: `yes`
- purpose: Verify WebAuthn registration response and rotate auth tokens.
- response: Access/refresh token payload with updated AAL context.
- https method: `POST`
- other: Frontend stores returned session via `supabase-js setSession`.

## 15) Sign Up with Passkey Modal (`/signup`, modal state)

Features:
- Passkey onboarding modal requires `full name` and `email`.
- Loading state while passkey ceremony is being prepared (Stitch loading variant).
- Success state after passkey enrollment (Stitch success variant).
- Uses existing Supabase Auth MFA WebAuthn routes (no custom passkey tables).
- If email already exists, frontend starts email OTP sign-in flow and then links passkey after authenticated session.

API contracts:
- route: `SUPABASE /auth/v1/signup`
- is protected: `no`
- purpose: Create account when email is new (provisional password + metadata).
- response: Session payload (autoconfirm) or pending confirmation state.
- https method: `POST`
- other: Frontend uses metadata `display_name` from modal full-name field.

- route: `SUPABASE /auth/v1/otp` (via `supabase-js signInWithOtp`)
- is protected: `no`
- purpose: Existing-email branch to prove ownership and continue linking to same identity.
- response: OTP/magic-link dispatch acknowledgement.
- https method: `POST`
- other: Used when signup returns existing-user error.

- route: `SUPABASE /auth/v1/factors`
- is protected: `yes`
- purpose: Enroll `factor_type=webauthn` for current signed-in user.
- response: Factor payload with `id` and status.
- https method: `POST`
- other: Uses Supabase Auth infrastructure directly and returns `mfa_webauthn_enroll_not_enabled` when WebAuthn MFA is not enabled on Auth service.

- route: `SUPABASE /auth/v1/factors/{factorId}/challenge`
- is protected: `yes`
- purpose: Generate WebAuthn registration challenge for factor.
- response: WebAuthn challenge payload with `credential_options`.
- https method: `POST`
- other: `webauthn.rpId` and `webauthn.rpOrigins` required.

- route: `SUPABASE /auth/v1/factors/{factorId}/verify`
- is protected: `yes`
- purpose: Verify WebAuthn credential response and finalize factor enrollment.
- response: Access/refresh token payload.
- https method: `POST`
- other: Frontend applies returned tokens with `supabase-js setSession`.
